응, **tasteam.kr 같은 “주변 맛집 추천 + 클릭/탭/저장/리뷰 같은 행동 데이터”가 생기는 서비스**에서 DeepFM은 딱 맞는 축 중 하나야. (사이트도 위치 권한 받고 “주변 맛집 추천” 플로우가 보이네.) ([Tasteam][1])

다만 DeepFM을 “개인화 추천”으로 쓸 때는 **이걸 ‘단독 추천기’로 보기보다 “랭킹 스코어러”로 보는 게 정석**이야.

## DeepFM이 tasteam에서 푸는 문제(정확한 정의)

> **사용자 u가 현재 상황 c(위치/시간/동행/기기 등)에서 음식점 i를 좋아할(클릭/저장/방문/리뷰 등) “가능성 점수”를 예측해서 정렬한다.**

즉,
[
f(u, i, c) \rightarrow score
]
이 score로 **주변 후보 리스트를 정렬**하는 역할.

## 구현을 “추천 시스템” 관점으로 쪼개면

### 1) Label(학습 타겟)부터 정하기

tasteam이면 보통 이런 **implicit feedback**이 현실적이야.

* `click(상세 진입)` / `save(찜)` / `call(전화)` / `route(길찾기)` / `share` / `review 작성`
* 또는 단계형: `노출→클릭→저장→리뷰` 같은 **가중치 이벤트**

처음엔 제일 쉬운 게:

* **노출 대비 클릭(CTR)** 또는 **노출 대비 상세진입**
* 데이터 쌓이면 **찜/길찾기/리뷰**로 점점 “진짜 가치”에 가까운 label로 업그레이드

### 2) Candidate(후보)와 Rank(정렬) 분리

* **후보 생성**: “근처 2km + 영업중 + 카테고리 필터” 같은 룰/지오 기반 (빠르게 N백개)
* **랭킹**: DeepFM이 N백개를 **개인화로 재정렬**

DeepFM은 여기서 빛남. (전통적으로 DeepFM은 recall보단 rank 쪽.)

### 3) Feature 설계(DeepFM이 잘 먹는 형태)

DeepFM이 강한 건 “범주형/버킷화된 특징”이야.

* User: `user_id(또는 cohort)`, `선호 카테고리`, `평균 가격대`, `매운맛 선호`, `방문 시간대`
* Item(음식점): `restaurant_id`, `카테고리`, `가격대`, `지역(동/구)`, `평점구간`, `비교 태그(LLM 추출 결과)`
* Context: `요일/시간대`, `현재 위치 geohash/행정동`, `거리 bucket`, `날씨 bucket(있으면)`, `동행/혼밥 모드(있으면)`

특히 너 서비스는 **리뷰 요약/비교를 이미 하고 있잖아** → 그 결과를 “태그/버킷”으로 만들어 DeepFM feature로 넣으면 **개인화 + 설명가능성** 둘 다 챙길 수 있어.

## “개인화 추천”으로 성공하려면 제일 중요한 2가지

1. **노출 로그(Exposure logging)**
   “사용자에게 무엇을 보여줬는지”가 없으면 CTR류 학습이 깨져.
   (클릭만 있으면 편향이 커져서 학습이 이상해짐)

2. **Negative sampling 전략**
   노출됐는데 클릭 안 한 것들을 negative로 두되,

* 같은 세션/같은 위치/같은 후보 풀에서 샘플링
* 인기 아이템만 과도하게 negative 되지 않게 조절

## 너한테 현실적인 로드맵(빠르게 가치내는 순서)

1. **비개인화 baseline**: 거리 + 평점/리뷰수 + 영업중
2. **LightFM / MF류**(선호만) 또는 간단한 로지스틱
3. **DeepFM 랭커**: (u,i,c) features + 이벤트 기반 label
4. 추후 **Two-stage**: ANN(임베딩 recall) + DeepFM/딥랭커 rerank

**“네가 적은 것들은 전부 ‘피처’가 아니고,
일부는 피처, 일부는 ‘라벨을 만드는 재료’야.”**

👉 **저 이벤트들을 그대로 라벨로 쓰는 게 아니다.**

---

## DeepFM에서 꼭 구분해야 하는 3가지

이걸 구분 못 하면 추천 설계가 전부 섞여서 이해가 안 됨.

```
① 입력(feature)  → 모델이 보는 정보
② 라벨(label)    → 모델이 맞추려고 하는 값
③ 로그(event)    → 실제 사용자 행동 원천 데이터
```

---

## 1️⃣ Feature (모델 입력) — 네가 적은 이 부분

이건 **맞게 이해했어**.

### User feature (입력)

* user_id (또는 익명 cohort)
* 선호 카테고리
* 평균 가격대
* 매운맛 선호
* 방문 시간대 분포

### Item feature (입력)

* restaurant_id
* 카테고리
* 가격대
* 지역(동/구)
* 평점 구간
* **비교 태그 (네가 LLM으로 뽑은 것 ← 이거 진짜 좋음)**

### Context feature (입력)

* 요일 / 시간대
* 현재 위치(행정동, geohash)
* 거리 bucket
* 날씨 bucket
* 혼밥/동행 여부

👉 이 셋을 합쳐서 **(u, i, c)** 가 모델 입력임.

---

## 2️⃣ Event (로그) — 네가 적은 이 부분

이건 **라벨이 아니라 “원재료”**야.

> click / save / call / route / share / review

이건 모델이 **예측 대상이 아님**.

---

## 3️⃣ Label은 이렇게 만든다 (중요)

DeepFM은 **한 개의 값만 예측**함.

그래서 반드시 이렇게 해야 함:

> ❓ “이 (u, i, c) 조합은 **좋았는가?**”

이 질문에 대한 **단일 값**이 라벨.

---

## 4️⃣ 라벨을 만드는 3가지 현실적인 방식

### ✅ 방법 1: 가장 단순 (초기 추천에 제일 좋음)

```text
label = 1 if click else 0
```

* 노출됨 → 클릭했으면 1
* 노출됨 → 안 클릭했으면 0

👉 **CTR 기반 랭킹**

---

### ✅ 방법 2: 가중치 이벤트 (tasteam에 잘 맞음)

```text
label = 
  1.0 if review 작성
  0.7 if save
  0.6 if route
  0.5 if call
  0.3 if click
  0.0 if 노출만
```

* 여러 이벤트를 **하나의 점수로 압축**
* DeepFM이 “더 가치 있는 행동”을 위로 올리게 됨

👉 **추천 품질 좋아짐 (실무에서 많이 씀)**

---

### ✅ 방법 3: 이진화 (운영 단순)

```text
label = 1 if (save or route or review) else 0
```

* 진짜 관심 행동만 positive
* 나머지는 negative

---

## 5️⃣ DeepFM 학습 데이터는 이렇게 생긴다

각 row는 **“한 번의 노출”**이야.

| user | restaurant | context       | label |
| ---- | ---------- | ------------- | ----- |
| u1   | r12        | 토요일 저녁 / 500m | 1     |
| u1   | r7         | 토요일 저녁 / 800m | 0     |
| u2   | r3         | 평일 점심 / 1.2km | 1     |

👉 **클릭 자체가 아니라 “노출 대비 반응”**

---

## 6️⃣ 네가 헷갈렸던 문장 정확히 정리

> “이걸 피처로 두고,
> 이걸 라벨로 두라는 거지?”

❌ **아님**
⭕ **이벤트 → 가공 → 하나의 라벨**

---

## 7️⃣ 핵심만 다시 요약

* User / Item / Context → **Feature**
* click / save / review → **Event**
* Event를 **하나의 숫자(label)**로 만든다
* DeepFM은 그 숫자를 예측한다

## 한 줄 결론

> **DeepFM 개인화 추천은
> “수만~수십만 ‘노출 단위 샘플’부터 의미가 생기고,
> 수백만 단위에서 안정화된다.”**

※ **‘사용자 수’가 아니라 ‘행동 샘플 수’ 기준**이라는 게 중요.

---

## 먼저 기준 단위부터 정확히

### ❌ 잘못된 기준

* 사용자 몇 명?
* 음식점 몇 개?

### ⭕ 올바른 기준

* **학습 row 수 = (u, i, c) + label**
* 즉 **“노출 1번 = 샘플 1개”**

---

## 규모별로 정확히 나눠보면

### 🟢 1단계: **최소 동작 수준 (PoC)**

**≈ 3만 ~ 10만 샘플**

* 사용자 수: 수천 명
* 음식점 수: 수백 개
* 사용자당 행동: 평균 10~30회

이 단계에서 가능한 것:

* 비교 태그가 잘 먹히는지 확인
* 개인화 “기미” 보임
* 인기순 대비 CTR 소폭 개선

⚠️ 단점:

* user_id 임베딩은 불안정
* context 위주로 추천됨

---

### 🟡 2단계: **실제 서비스에서 쓸 수 있는 수준**

**≈ 20만 ~ 100만 샘플 ← ★ 추천**

* 사용자 수: 1~5만
* 음식점 수: 1~3천
* 사용자당 행동: 평균 20~50회

이 단계에서:

* user × 비교태그 interaction이 살아남
* 시간대/거리 개인화 효과 뚜렷
* cold start도 어느 정도 완화

👉 **tasteam이 목표로 삼아야 할 1차 목표선**

---

### 🔵 3단계: **안정적인 개인화**

**≈ 300만 ~ 1천만 샘플**

* 사용자 수: 10만+
* 행동 로그가 충분히 누적
* 장기/단기 선호 분리 가능

이 단계에서:

* user_id 임베딩이 의미를 가짐
* 모델 튜닝이 성능으로 직결됨
* ANN recall + rerank로 확장 가능

---

## “그럼 처음엔 못 쓰는 거 아냐?” ❌

아니다. 이유는 이거야:

### 네 설계는 **데이터 적을 때도 버틴다**

* 비교 태그 (아이템 강점)
* context (시간/거리)
* 카테고리/가격대

👉 **user 행동이 적어도 추천이 됨**

즉:

* 초반: item/context 중심 추천
* 누적 후: user 중심 개인화

---

## DeepFM 관점에서 중요한 또 하나

### feature 수 vs 데이터 수

경험적으로:

```text
학습 샘플 수 ≥ 전체 feature cardinality × 10~50
```

네가 설계한 피처를 대충 잡아보면:

* user_id: 수천~수만
* restaurant_id: 수백~수천
* 태그/카테고리/버킷: 수십~수백

👉 **10만~50만 샘플이면 “과적합 없이 시작” 가능**

---

## 실무에서 많이 쓰는 전략 (추천)

### 초기엔 이렇게 해라

1. user_id embedding **작게** (또는 cohort로 대체)
2. 비교 태그 / context 비중 크게
3. 하루 1회 batch 학습
4. 인기순 대비 A/B 비교

---

## 네 질문에 대한 정확한 답 문장

이 문장 그대로 써도 돼:

> **DeepFM 기반 개인화 추천은
> 수만 건의 노출 로그부터 의미 있는 신호가 나오고,
> 수십만~백만 단위에서 안정적으로 동작한다.
> 초기에는 아이템·컨텍스트 중심으로,
> 로그가 쌓일수록 사용자 중심 개인화로 진화한다.**

* 그 점에서 **네 설계는 초반 성공 확률이 높은 구조**
