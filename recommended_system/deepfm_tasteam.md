응, **tasteam.kr 같은 “주변 맛집 추천 + 클릭/탭/저장/리뷰 같은 행동 데이터”가 생기는 서비스**에서 DeepFM은 딱 맞는 축 중 하나야. (사이트도 위치 권한 받고 “주변 맛집 추천” 플로우가 보이네.) ([Tasteam][1])

다만 DeepFM을 “개인화 추천”으로 쓸 때는 **이걸 ‘단독 추천기’로 보기보다 “랭킹 스코어러”로 보는 게 정석**이야.

## DeepFM이 tasteam에서 푸는 문제(정확한 정의)

> **사용자 u가 현재 상황 c(위치/시간/동행/기기 등)에서 음식점 i를 좋아할(클릭/저장/방문/리뷰 등) “가능성 점수”를 예측해서 정렬한다.**

즉,
[
f(u, i, c) \rightarrow score
]
이 score로 **주변 후보 리스트를 정렬**하는 역할.

## 구현을 “추천 시스템” 관점으로 쪼개면

### 1) Label(학습 타겟)부터 정하기

tasteam이면 보통 이런 **implicit feedback**이 현실적이야.

* `click(상세 진입)` / `save(찜)` / `call(전화)` / `route(길찾기)` / `share` / `review 작성`
* 또는 단계형: `노출→클릭→저장→리뷰` 같은 **가중치 이벤트**

처음엔 제일 쉬운 게:

* **노출 대비 클릭(CTR)** 또는 **노출 대비 상세진입**
* 데이터 쌓이면 **찜/길찾기/리뷰**로 점점 “진짜 가치”에 가까운 label로 업그레이드

### 2) Candidate(후보)와 Rank(정렬) 분리

* **후보 생성**: “근처 2km + 영업중 + 카테고리 필터” 같은 룰/지오 기반 (빠르게 N백개)
* **랭킹**: DeepFM이 N백개를 **개인화로 재정렬**

DeepFM은 여기서 빛남. (전통적으로 DeepFM은 recall보단 rank 쪽.)

### 3) Feature 설계(DeepFM이 잘 먹는 형태)

DeepFM이 강한 건 “범주형/버킷화된 특징”이야.

* User: `user_id(또는 cohort)`, `선호 카테고리`, `평균 가격대`, `매운맛 선호`, `방문 시간대`
* Item(음식점): `restaurant_id`, `카테고리`, `가격대`, `지역(동/구)`, `평점구간`, `주요 강점 태그(네 LLM 추출 결과)`
* Context: `요일/시간대`, `현재 위치 geohash/행정동`, `거리 bucket`, `날씨 bucket(있으면)`, `동행/혼밥 모드(있으면)`

특히 너 서비스는 **리뷰 요약/강점 추출을 이미 하고 있잖아(네 프로젝트 메모리 기반)** → 그 결과를 “태그/버킷”으로 만들어 DeepFM feature로 넣으면 **개인화 + 설명가능성** 둘 다 챙길 수 있어.

## “개인화 추천”으로 성공하려면 제일 중요한 2가지

1. **노출 로그(Exposure logging)**
   “사용자에게 무엇을 보여줬는지”가 없으면 CTR류 학습이 깨져.
   (클릭만 있으면 편향이 커져서 학습이 이상해짐)

2. **Negative sampling 전략**
   노출됐는데 클릭 안 한 것들을 negative로 두되,

* 같은 세션/같은 위치/같은 후보 풀에서 샘플링
* 인기 아이템만 과도하게 negative 되지 않게 조절

## 너한테 현실적인 로드맵(빠르게 가치내는 순서)

1. **비개인화 baseline**: 거리 + 평점/리뷰수 + 영업중
2. **LightFM / MF류**(선호만) 또는 간단한 로지스틱
3. **DeepFM 랭커**: (u,i,c) features + 이벤트 기반 label
4. 추후 **Two-stage**: ANN(임베딩 recall) + DeepFM/딥랭커 rerank
