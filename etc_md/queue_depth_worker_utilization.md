## Queue depth vs Worker utilization

| | Queue depth (`app_queue_depth`) | Worker utilization (`app_worker_busy`) |
|--|----------------------------------|----------------------------------------|
| **의미** | (이 워커/프로세스에서) **지금 처리 중인 요청 개수** | 이 워커가 **하나라도 처리 중이면 1, 아니면 0** |
| **값 범위** | 0, 1, 2, 3, … (제한 없음) | 0 또는 1 |

---

## 언제 같고 언제 다른지

- **싱글 워커 + 동기**  
  한 번에 요청 1개만 처리하면  
  - `queue_depth` = 0 또는 1  
  - `worker_busy` = 0 또는 1  
  → 이 경우엔 **숫자가 같아서** “지금 바쁜지 여부”만 보면 둘 다 비슷하게 느껴질 수 있음.

- **싱글 워커 + 비동기**  
  한 워커가 여러 요청을 동시에 처리하면  
  - `queue_depth` = 2, 5, 10 … (동시 처리 수)  
  - `worker_busy` = 1 (한 건이라도 있으면 1)  
  → **활용률은 100%인데, queue depth는 “몇 개나 쌓였는지”**를 더 보여줌.

- **멀티 워커**  
  - 대시보드 **Worker utilization** = `sum(app_worker_busy) / count(워커)` → **“워커 중 몇 %가 바쁜가”** (0~1).  
  - **Queue depth** = 보통 **전체(또는 인스턴스별) in-flight 요청 수** 합.  
  → 예: 워커 4개, 3개만 바쁨 → utilization = 75%, queue depth = 3 (또는 더 많을 수 있음).  
  → **“몇 % 워커가 쓰이는지”**와 **“몇 개 요청이 동시에 처리 중인지”**는 다른 지표.

---

**정리:**  
- **Queue depth** = 동시 처리 중인 **요청 개수** (수치 그대로, async면 1보다 클 수 있음).  
- **Worker utilization** = **워커가 바쁜 비율** (0~1, “몇 % 워커가 일하고 있는지”).  

그래서 “워커 활용률 = 현재 queue depth”라고 보기보다는, **“워커 활용률은 ‘바쁜 워커 비율’이고, queue depth는 ‘동시 처리 중인 요청 수’”**로 보는 게 맞습니다.
