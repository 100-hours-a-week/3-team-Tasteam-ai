Q. 배치 실행 주체를 API에서 분리 같은 코드(파이프라인)는 재사용하되, 실행은 batch-runner 컨테이너(커맨드 실행형) 또는 워커(Celery/RQ/SQS 소비자) 로 분리하면 “배치 때문에 API가 흔들리는” 문제가 줄어듦. 이게 무슨말인지 모르겟음 왜 분리하면 배치때문에 API가 흔들리지 않지?

A. 지금 구조에선 **API 서버가 “요청 받는 역할” + “배치(무거운 작업) 실행자 역할”을 동시에** 하고 있어.
그래서 배치가 돌면 API가 흔들려.

“분리”는 **무거운 실행을 API 프로세스 밖(다른 컨테이너/다른 서버/다른 워커 프로세스)** 로 빼는 거고, 그러면 흔들릴 원인이 API 안에 없어져.

## 1) 지금처럼 API 안에서 배치를 돌리면 왜 흔들리냐

배치가 하는 일은 보통 이런 것들이야:

* **CPU/메모리 많이 먹음** (리뷰 로딩/전처리/벡터서치/결과 합치기)
* **네트워크 커넥션 많이 씀** (DB, OpenAI, 벡터DB)
* **동시 요청 많이 띄움** (`asyncio.gather`, 병렬 처리)
* **오래 걸림** (수분~수시간)
* **429 나오면 재시도/대기**가 생김

이게 API 프로세스 안에서 일어나면:

* 이벤트 루프/스레드/커넥션 풀이 배치에 점유됨
* 메모리 증가 → GC/스와핑/심하면 OOM
* 외부 호출이 몰리면서 전체 지연 증가
* 결국 **평범한 사용자 요청(짧아야 하는 API 호출)** 도 같이 느려지거나 타임아웃남

즉, **리소스를 “같은 통”을 쓰기 때문에** 흔들리는 거야.

## 2) “분리”가 정확히 뭐냐 (예시로 감 잡기)

### 분리 전 (지금 흔한 형태)

* 사용자/스케줄러 → **API 서버**
* API 서버가 내부에서 바로 “배치 함수” 실행
* API 서버가 배치 동안 CPU/메모리/커넥션 다 잡아먹음
  → API 서버 자체가 느려짐/죽음

### 분리 후

* 사용자/스케줄러 → **API 서버**
* API 서버는 “작업 하나 만들고” **큐에 넣기만 함** (몇 ms~수십 ms)
* **워커(별도 컨테이너)** 가 큐에서 작업을 가져가 실행
* 결과는 DB/S3에 저장
  → API 서버는 계속 가볍게 요청-응답만 처리

핵심: **API와 배치가 서로 다른 프로세스/컨테이너/서버의 자원(CPU/메모리/커넥션)을 쓰게 만드는 것**.

## 3) 왜 분리하면 “배치 때문에 API가 흔들리지” 않나

한 줄로: **배치가 폭주해도 API가 쓰는 자원이 별개라서**.

* 배치 워커가 CPU 100% 먹어도 → API 컨테이너 CPU는 그대로
* 배치 워커가 메모리 터져서 죽어도 → API 프로세스는 안 죽음
* 배치 워커가 OpenAI 429로 재시도 대기 걸려도 → API는 그냥 job 상태만 보여주면 됨
* 배치 워커가 DB 커넥션을 많이 쓰면 → (설계를 잘하면) 워커 전용 풀/제한으로 API 풀을 보호 가능

즉, **격리(isolation)** 가 되는 거야.

## 4) 비유로 이해하면 더 쉬움

* **API 서버 = 카운터 직원(주문 받기)**
* **배치/워커 = 주방(요리하기)**

주문 받는 직원이 주방에서 요리까지 하면,

* 손님이 몰리면 주문도 못 받고 요리도 못 해서 매장 전체가 마비됨.

주문은 카운터가 받고,
요리는 주방이 하면,

* 주방이 밀려도 카운터는 주문을 계속 받고 “대기번호”를 줄 수 있음.

큐(job_id)가 **대기번호표** 역할.

## 5) 너한테 딱 맞춘 결론

너 프로젝트에서 배치가 흔들리는 대표 원인이:

* OpenAI 429 + 재시도/대기
* asyncio 병렬 처리로 커넥션/메모리/이벤트루프 점유
* 장시간 작업

이걸 **API 프로세스 밖으로 빼면**, API는

* “작업 등록”
* “결과 조회”
  만 해서 안정적으로 유지돼.

원하면, 네 현재 구조(이미 batch 엔드포인트 있음) 기준으로 **“API는 job 만들기/조회만 남기고, 기존 batch 함수는 워커로 옮기는”** 최소 변경 버전을 바로 그림처럼 정리해줄게.

Q. 프로세스를 분리한다는거야?

A. 응.
정확히 말하면 **“프로세스를 분리한다”**는 뜻이 맞아.
더 정확히는 **“실행 프로세스(= 런타임 자원)를 분리한다”**야.

조금 더 구조적으로 정리해줄게.

---

# 1️⃣ 지금 구조 (분리 안 된 상태)

```
[FastAPI 프로세스]
    ├─ HTTP 요청 처리
    ├─ DB 조회
    ├─ 벡터서치
    ├─ LLM 호출
    ├─ asyncio 병렬 처리
    └─ 재시도 / 대기
```

👉 **모든 게 한 프로세스 안에서 실행됨**

문제는:

* 배치가 CPU/메모리 먹으면 → API 응답 느려짐
* LLM 429 재시도 대기 → 이벤트 루프 점유
* 대량 병렬 처리 → 커넥션 풀 고갈
* 메모리 증가 → OOM 나면 서버 전체 죽음

즉, **같은 프로세스 = 같은 자원 풀을 공유**

---

# 2️⃣ 분리된 구조

```
[FastAPI 프로세스]             [Worker 프로세스]
    ├─ HTTP 요청 처리    →       ├─ 벡터서치
    ├─ 입력 검증         →       ├─ LLM 호출
    ├─ job 생성          →       ├─ 재시도/백오프
    └─ job_id 반환       →       ├─ 결과 저장
                                  └─ 상태 업데이트
```

👉 서로 다른 프로세스
👉 서로 다른 CPU/메모리/이벤트루프
👉 서로 다른 커넥션 풀

이게 핵심이야.

---

# 3️⃣ 왜 “프로세스 분리”가 안정성을 주는가

운영에서 제일 중요한 개념이 하나 있어:

> **Isolation (격리)**

격리가 되면:

* Worker가 CPU 100% 써도 → API는 정상
* Worker가 메모리 터져도 → API는 안 죽음
* Worker가 OpenAI 429로 2분 대기해도 → API는 그냥 job 상태만 보여주면 됨

---

# 4️⃣ 컨테이너 분리 vs 프로세스 분리

실제로는 보통:

* API 컨테이너 1개
* Worker 컨테이너 1개 이상

처럼 **컨테이너 단위로 분리**함.

하지만 본질은 컨테이너가 아니라:

> **운영체제 레벨에서 다른 프로세스인가?**
> **자원 풀을 공유하는가?**

이거야.

---

# 5️⃣ 아주 현실적인 예

만약 지금 네 FastAPI 서버에서:

```python
await asyncio.gather(*1000개의 LLM 호출)
```

이게 API 프로세스 안에서 실행되면?

* 그 순간 API 서버는 사실상 배치 서버가 됨
* 다른 사용자 요청이 와도 처리 밀림

하지만 그 코드가 Worker 프로세스에서 실행되면?

* API는 그냥:

  ```python
  publish_job()
  return job_id
  ```

  이거만 함 (수 ms)

완전히 다름.

Q. 비동기가 요청을 보내서 요청을 보낸 서버의 메모리를 효율적으로 사용하고, OOM을 방지한다는걸로 아는데, 지금처럼 프로세스를 분리했을때 이야기인가?

A. 맞아, 네가 알고 있는 “비동기 = 메모리 효율/ OOM 방지에 도움”이라는 말은 **일부 상황에만** 맞고, 지금 우리가 말하는 **“프로세스 분리”랑은 다른 축**이야. 둘은 **겹치기도 하지만 동일하지 않아.** (네 아키텍처에서 배치가 `asyncio.gather`로 레스토랑 단위/파이프라인 내부 단위 병렬을 하는 구조라는 전제가 있어.) 

## 1) 비동기(async)가 OOM을 “자동으로” 막아주진 않아

비동기는 본질적으로 **I/O 대기 시간을 겹쳐서 처리량을 올리는 방식**이야.
즉 “메모리 절약”이 1차 목적은 아니고, 오히려 이런 경우엔 **메모리가 더 터질 수 있어**:

* `asyncio.gather`로 작업을 한꺼번에 많이 띄움 → **동시에 살아있는 태스크/응답 객체/버퍼가 늘어남**
* 각 태스크가 리뷰 리스트를 들고 있거나(대량 payload), LLM 응답을 모아서 후처리하면 → **피크 메모리 증가**
* 외부 호출이 느려져서 태스크가 오래 살아있으면 → **동시 생존 개수 증가 → 메모리 증가**
* 그래서 너도 이미 세마포어로 동시성 제한(BATCH_*_CONCURRENCY) 걸어두는 거고, 이게 OOM 방지의 핵심 장치야. 

✅ 정리: **비동기 자체가 OOM을 막는 게 아니라, “동시성 제한 + 스트리밍/청크 처리 + 결과를 빨리 버리기”가 OOM을 막아.**

## 2) “프로세스 분리”는 OOM을 막는 방식이 다르다 (격리)

프로세스 분리는 “메모리를 효율적으로” 쓰는 게 아니라, **터져도 API가 같이 안 죽게 하는 것(격리)**이 핵심이야.

* 배치를 API 프로세스 안에서 돌리면: 배치가 메모리 터짐(OOM) → **API 프로세스가 죽음** → 서비스 다운
* 배치를 워커 프로세스로 분리하면: 배치 워커가 메모리 터짐 → **워커만 죽음** → API는 계속 살아있고, 재시도/재기동으로 복구 가능

즉, 프로세스 분리는

* **“OOM이 발생할 수 있는 영역”을 API 밖으로 빼서**
* **OOM의 피해 범위를 줄이는 전략**이야.

✅ 정리: **비동기 = (잘 쓰면) 한 프로세스 안에서 효율/처리량**
**프로세스 분리 = 한 프로세스가 터져도 다른 프로세스는 안 터지게 격리**

## 3) 네 상황에서 정확히 어디가 헷갈린 포인트냐

너 아키텍처 문서에도 “배치 API에서 2수준 비동기(레스토랑 간, 파이프라인 내)”가 있다고 되어 있지? 
이건 “API 서버 프로세스 안에서” 비동기로 배치를 돌리는 모델이야.

여기서 OOM 방지의 관건은:

* **동시성(세마포어) 제한** 
* **한 번에 메모리에 올리는 리뷰 양 제한(limit, per_category_max 등)** 
* **중간 결과 빨리 해제/저장(스트리밍/청크)**

반면 “프로세스 분리”를 하면:

* 위의 것들이 잘못돼서 워커가 OOM 나도,
* API는 안 죽고 “job 실패”로만 남게 만들 수 있어.

## 4) 결론

* 네가 말한 “비동기 = OOM 방지”는 **‘동시성 제한을 잘 건 비동기’일 때만** 어느 정도 성립해.
* 우리가 말한 “프로세스 분리”는 **OOM 자체를 없애는 게 아니라, OOM의 폭발 반경을 줄여서 API를 보호**하는 이야기야.
* 그래서 실전에서는 보통 둘 다 씀:

  * 워커 내부에서도 **비동기 + 세마포어**로 처리량 뽑고 
  * API는 그 워커와 **프로세스 분리**로 안정성을 확보

원하면, 너 현재 설정들(`SUMMARY_RESTAURANT_ASYNC`, `BATCH_LLM_CONCURRENCY` 등) 기준으로 **“어느 지점에서 메모리 피크가 생기는지”**를 딱 찍어서(레스토랑 수/리뷰 수/limit 조합) 안전한 동시성 가이드를 바로 잡아줄게.
