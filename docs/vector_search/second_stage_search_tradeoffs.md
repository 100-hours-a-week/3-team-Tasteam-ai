# 2차 검색(Hybrid/RRF·Rerank) 조건 선택 트레이드오프

Summary 등에서 **1차 Dense-only → 필요 시 2차 Hybrid/RRF(또는 rerank)** 를 쓸 때,  
**“언제 2차를 켤지”** 에 대한 조건을 어떻게 정할지의 **선택 관점별 트레이드오프** 정리.

---

## 1) 결과 개수(coverage) 기반

| 항목 | 내용 |
|------|------|
| **조건 예** | `hits < k_min` / 카테고리별 균형 깨짐(service만 0, price/food만 많음) → 그 카테고리만 2차 |
| **장점** | 구현 단순, “빈 결과”를 확실히 줄임. 0건·극소건에서 즉시 개선. |
| **단점** | “개수만” 보므로, 3~5건 있어도 전부 비관련이면 품질은 나쁨. 오탐이 많아도 2차를 안 켤 수 있음. |
| **정확도** | Recall(빈 카테고리 방지)에는 강함. Precision(상위가 진짜 관련)은 보장 못 함. |
| **비용/지연** | 2차 호출 빈도는 “부족한 케이스”에만 한정 → 중간. |
| **선택 시점** | **0건/극소건이 자주 나오는 서비스**에서 우선 적용. “언급이 적다” 문제 해소에 적합. |

---

## 2) 점수 분포/신뢰도 기반 (Dense score signals)

| 항목 | 내용 |
|------|------|
| **조건 예** | `max_score < threshold` / `top1 - topN` 작음(랭킹 불안정) / 동점·비슷한 점수 후보 많음 → 2차 |
| **장점** | “의미적 유사도”가 애매할 때만 2차를 켜서, 불필요한 2차 호출을 줄임. Dense-only 단계 점수는 해석 가능. |
| **단점** | threshold·N 선택이 데이터/모델에 의존. RRF 사용 시 2차 결과 점수는 해석이 어려울 수 있음. |
| **정확도** | “애매한 랭킹”을 2차로 정교화 → Precision 개선에 유리. Recall은 1차에 맡김. |
| **비용/지연** | 조건이 잘 맞으면 2차 호출을 필요한 경우로만 제한 → 비용 대비 효과 좋음. |
| **선택 시점** | **1차에서 후보는 충분한데 상위가 자주 어색한** 경우. 점수 로그가 쌓이면 threshold 튜닝 가능. |

---

## 3) 텍스트 기반 간단 필터(cheap heuristic)

| 항목 | 내용 |
|------|------|
| **조건 예** | 카테고리 키워드 hit 거의 없음 / 상위가 너무 짧은 리뷰만 / 중복(유사 문장) 심함 → 2차 |
| **장점** | 도메인(카테고리)과 직접 연결. “의미는 비슷한데 키워드 없음” 같은 오탐을 걸러낼 수 있음. |
| **단점** | 키워드 리스트·짧음/중복 기준을 유지해야 함. 신조어·표현 다양성에 약할 수 있음. |
| **정확도** | 카테고리별 **적합도(키워드)** 로 필터 → Precision 향상. 키워드 설계가 좋으면 강함. |
| **비용/지연** | 텍스트만 스캔하므로 CPU 부하는 작음. 2차 호출 여부만 결정 → 저비용. |
| **선택 시점** | **특정 카테고리(예: service/price)에서 오탐이 패턴으로 보일 때**. 규칙을 도메인에 맞게 갱신 가능할 때. |

---

## 4) 쿼리/요청 특성 기반

| 항목 | 내용 |
|------|------|
| **조건 예** | 쿼리가 길거나 복합 의도 / 특정 카테고리(가격 등)는 오탐이 잦으면 기본적으로 2차 |
| **장점** | “어려운 쿼리”나 “문제 카테고리”에만 비용을 써서, 전체 품질을 골고루 올리기 좋음. |
| **단점** | “길다/복합이다” 정의·카테고리별 플래그를 설계·운영해야 함. 과한 2차 호출 가능. |
| **정확도** | 어려운 쿼리·취약 카테고리의 Precision을 선택적으로 올림. |
| **비용/지연** | 특정 카테고리/쿼리 타입에만 2차 → 전역보다는 비용 통제 가능. |
| **선택 시점** | **가격 등 특정 카테고리에서 오탐이 반복**될 때, 또는 **복합 쿼리 비율이 높을 때**. |

---

## 5) 비용 대비 효과(운영) 기준

| 항목 | 내용 |
|------|------|
| **조건 예** | `hits < 5` 또는 `max_score < th` 일 때만 2차 / 상위 N개만 rerank(예: 50→20) |
| **장점** | 2차 호출 횟수·대상 개수를 제한해 지연·비용을 예측 가능하게 함. |
| **단점** | “정확도만 보면 2차가 유리한데” 비용 때문에 스킵하는 케이스 생김. |
| **정확도** | 제한을 많이 걸수록 정확도 상한이 낮아짐. N을 키우면 비용↑. |
| **비용/지연** | 2차 빈도·rerank N으로 직접 제어 → 운영 관점에서 가장 명확. |
| **선택 시점** | **트래픽·비용 제약이 있을 때**. 먼저 1~4로 조건을 정한 뒤, 5로 상한(cap)만 둘 때. |

## 권장 조합(정확도 우선 가정)

- **정확도만 고려**할 때:
  - **1) + 2) + 3)** 을 함께 쓰는 것이 가장 균형 좋음.
  - 1): `hits < k_min` 또는 카테고리 균형 깨짐 → 2차로 **coverage** 확보.
  - 2): `max_score` 낮음 / 점수 격차 작음 → 2차로 **랭킹 신뢰도** 보완.
  - 3): 카테고리 키워드 적합도 낮음 / 짧은 리뷰만 / 중복 많음 → 2차로 **도메인 적합도** 보완.
- **비용·지연**이 이슈가 되면 **5)** 로 “2차 상한”만 추가(예: 2차 연산은 상위 N개만, 또는 2차 호출은 카테고리당 1회로 제한).

현재 Summary 경로의 `_retrieve_category_hits_accuracy_first` 는 **1) + 2) + 3)** 을 반영하고, 부족 시 넓은 쿼리·최근 리뷰 폴백으로 **coverage** 를 보완하는 구조로 맞춰 두었음.
