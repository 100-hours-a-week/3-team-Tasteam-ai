## 1. 각 모델 특성

| 모델 | 현재 구현 | GPU 효과 | 호출 빈도 |
|------|-----------|----------|-----------|
| **Sentiment** | HuggingFace pipeline | 높음 (추론 속도 ↑) | 중간 |
| **LLM** | vLLM / OpenAI / RunPod | 필수 (7B+ 크기) | 낮음 |
| **Embedding** | FastEmbed (ONNX 기반) | 낮음 (CPU에서도 빠름) | 매우 높음 |

---

## 2. GPU 서버로 모두 올릴 때의 이점

- **FastAPI를 CPU 전용으로 유지**  
  - PyTorch/transformers/vLLM 등 무거운 의존성 제거  
  - 이미지 경량화, 시작 속도 개선

- **GPU 리소스 집중**  
  - 모든 추론을 하나의 GPU 서버에서 관리  
  - GPU 메모리/연산 효율적으로 사용 가능

- **배포 단순화**  
  - CPU 서버: API·라우팅·비즈니스 로직  
  - GPU 서버: 모든 모델 추론  
  - 역할 분리가 명확해짐

---

## 3. 주의할 점

### 3.1 임베딩은 CPU에서도 충분할 수 있음

- FastEmbed는 ONNX 기반이라 CPU에서도 이미 빠릅니다.  
- GPU로 옮겨도 이득이 크지 않을 수 있고, 네트워크 왕복이 오히려 비용이 됩니다.

### 3.2 임베딩 호출 빈도와 지연 시간

임베딩은 호출이 잦습니다.

- 벡터 검색: **쿼리마다** `encoder.encode(query_text)` 1회
- 리뷰 인덱싱: **리뷰마다** 인코딩
- Summary/Comparison: 내부에서 여러 번 인코딩

이걸 전부 원격 GPU 서버로 보내면:

- 호출당 네트워크 지연이 더해짐 (예: 5~20ms)
- 배치 API를 잘 설계하면 완화 가능 (요청 1번에 여러 문장 인코딩)

### 3.3 단일 장애점

임베딩·감성·LLM을 한 GPU 서버에 모으면:

- GPU 서버 장애 시:
  - 벡터 검색 불가 (쿼리 인코딩 불가)
  - 감성 분석 불가
  - 요약·비교 불가  
- FastAPI는 살아 있어도 실제 사용이 거의 불가능해짐

LLM만 분리했을 때보다 영향 범위가 넓어집니다.

### 3.4 GPU 메모리

대략:

- Sentiment: ~200–500MB
- Embedding (Dense+Sparse): ~500MB–1GB
- LLM 7B (fp16): ~14GB 이상  
→ 16–24GB GPU 한 장으로는 부족할 수 있고, 멀티 GPU나 모델 로딩 전략이 필요합니다.

## 4. 추천 구성

### 옵션 A: LLM + Sentiment만 GPU 서버 (임베딩은 CPU 유지)

```
[FastAPI CPU]
  - 임베딩 (FastEmbed) ← CPU, 호출 빈도 높음
  - 벡터 검색, 라우팅

[GPU 서버]
  - LLM
  - Sentiment
```

- 임베딩은 CPU에서 그대로 두고, LLM과 Sentiment만 GPU로 분리  
- 장점:  
  - 벡터 검색 지연 시간 영향 적음  
  - GPU 메모리 부담 감소  
  - GPU 다운 시에도 벡터 검색은 가능  
- 단점: FastAPI 이미지에 FastEmbed는 유지해야 함 (이미 가벼운 편)

### 옵션 B: 세 가지 모두 GPU 서버

```
[FastAPI CPU]
  - 벡터 검색 로직, 라우팅
  - 쿼리 인코딩은 원격 API 호출

[GPU 서버]
  - LLM
  - Sentiment
  - Embedding
```

- 장점:  
  - FastAPI를 완전히 모델 없이 유지  
  - GPU 인프라 한 곳에서 관리  
- 단점:  
  - 벡터 검색 지연 시간 증가  
  - 임베딩 API 배치 설계 필요  
  - GPU 장애 시 전체 기능 중단

### 옵션 C: 단계적 도입

1. 1단계: **LLM**만 GPU 서버로 분리 (효과 큼)  
2. 2단계: **Sentiment**까지 GPU로 이전  
3. 3단계: 임베딩도 GPU로 옮길지, 벡터 검색 지연 시간을 보고 결정  

---

## 5. 정리

- **LLM** → GPU 서버로 분리하는 것은 매우 합리적입니다.  
- **Sentiment** → GPU 서버로 옮기는 것도 성능 측면에서 유리합니다.  
- **Embedding** → GPU로 옮기는 것은 선택 사항입니다.  
  - FastEmbed는 CPU에서도 충분히 빠르고  
  - 호출이 많아서 네트워크 비용·지연이 문제될 수 있습니다.

가장 균형 있는 선택은 **옵션 A(LLM + Sentiment만 GPU)**이고,  
“API 서버를 최대한 가볍게”를 우선한다면 **옵션 B**도 가능하되, 임베딩 API는 반드시 **배치 API**로 설계하는 것이 좋습니다.
