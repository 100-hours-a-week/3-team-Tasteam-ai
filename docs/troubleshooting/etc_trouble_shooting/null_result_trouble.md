‘쿼리로 서치가 안 돼서 리뷰가 0개(또는 매우 적음) → LLM이 “언급이 적다”만 내는 문제’를 줄이는 방향만 정리했습니다. (Ask 모드라 코드 수정은 하지 않았습니다.)

---

### 1. **검색 단계에서 보완 (리뷰를 더/다르게 가져오기)**

- **카테고리별 시드 확장**  
  - `DEFAULT_SERVICE_SEEDS` 등에 키워드를 더 넣어서, 해당 카테고리와 관련된 리뷰가 더 걸리게 함.
- **prefetch 늘리기**  
  - `dense_prefetch_limit` / `sparse_prefetch_limit`를 키우면 RRF 후보가 많아져, 일부 카테고리에서라도 더 많이 걸릴 수 있음.
- **카테고리 0건일 때만 폴백 검색**  
  - 어떤 카테고리(예: service)가 0건이면, **그 카테고리만**  
    - 같은 `restaurant_id`로 **더 넓은 쿼리**(예: "서비스 음식 가격 만족")로 한 번 더 검색하거나,  
    - 또는 **해당 레스토랑 리뷰 N개**를 시드 없이(또는 매우 짧은 쿼리로) 가져와서  
  - 그 결과를 그 카테고리 입력으로 넣어 주면, “완전 빈 리스트” 대신 뭔가 들어가서 “언급이 적다”만 나오는 상황을 줄일 수 있음.

---

### 2. **파이프라인에서 “0건”을 명시적으로 처리**

- **카테고리별로 검색 결과가 0건이면**  
  - LLM에 **빈 배열을 그대로 넘기지 말고**,  
    - “이 카테고리로는 검색된 리뷰가 없음”을 문장으로 넣거나,  
    - 아예 그 카테고리용 **고정 문구**를 정해 두는 방식.
- 예:  
  - 해당 카테고리 summary를 **코드에서 고정**  
    - `"이 카테고리로 검색된 리뷰가 없어 요약할 수 없습니다."`  
  - 또는 프롬프트에만  
    - “리스트가 비어 있으면 summary는 반드시 ‘이 카테고리로 검색된 리뷰가 없습니다.’로 써라”라고 명시.

이렇게 하면 “쿼리로 서치를 못한 경우”를 **“언급이 적다”가 아니라 “검색 결과 없음”**으로 명확히 구분할 수 있습니다.

---

### 3. **폴백으로 “전체 리뷰” 일부 사용**

- 카테고리별 하이브리드 검색이 **0건인 카테고리만** 골라서,  
  - 그 레스토랑의 **최근 리뷰 N건**(벡터 검색의 `restaurant_id` 필터만 걸어서 가져오기 등)을 넣어 주는 방법.
- 프롬프트에  
  - “이 카테고리와 직접 매칭된 리뷰는 없고, 아래는 해당 레스토랑의 최근 리뷰 일부이다. 가능한 범위에서만 요약하라.”  
  같은 설명을 넣으면, LLM이 “언급이 적다” 대신 “전체 리뷰 기준으로 보면 …” 같은 요약을 할 수 있음.
- 다만 카테고리 의미가 흐려질 수 있어서, **service/price/food 중 0건인 것에만** 제한적으로 쓰는 게 안전함.

---

### 4. **프롬프트만 조정**

- “근거(입력 리뷰)에 없으면 **‘언급이 적다’**라고 표현하라”를  
  - “**해당 카테고리로 검색된 리뷰가 없으면** summary는 ‘이 카테고리로 검색된 리뷰가 없습니다.’로 써라. **리스트가 비어 있으면** 추측하지 말고 이 문구만 사용하라.”  
  처럼 바꾸는 방법.
- 구현이 가장 가볍고, “쿼리로 서치를 못한 경우”와 “리뷰는 있는데 해당 주제 언급이 적은 경우”를 구분하는 데 도움이 됨.

이렇게 하면 “쿼리로 서치를 못한 리뷰들”에 대해서는 “언급이 적다” 대신 **“검색된 리뷰 없음”** 또는 **폴백 리뷰 기반 요약**으로 정리할 수 있습니다. 원하시는 방향(프롬프트만 / 0건 시 고정 문구 / 폴백 검색 추가)을 정해 주시면, 그에 맞춰 구체적인 수정 위치(파일·함수)까지 짚어 드리겠습니다.

---

“0건인 카테고리에 한해 폴백 검색을 한 번 더 한다”는 말은, **카테고리별 벡터 검색 결과가 비었을 때(=리뷰를 못 가져왔을 때)** 그 카테고리만 **두 번째 방식으로 다시 리뷰를 채워 넣자**는 뜻입니다.

## 1) 현재 흐름 (문제 발생 케이스)

- Summary는 카테고리별(service/price/food)로 **각각 벡터 검색**을 합니다.
- 예를 들어 “service” 검색이 **0건**이면 LLM 입력이 이렇게 됩니다:

```json
{
  "service": [],
  "price": ["...리뷰들..."],
  "food": ["...리뷰들..."]
}
```

그러면 LLM은 규칙대로 “언급이 적다/정보가 부족” 같은 말만 하게 됩니다.

## 2) 제안한 “폴백 검색”이란?

**service가 0건인 그 순간에만**, service를 빈 배열로 두지 말고 “대체 리뷰”를 가져와서 채우자는 겁니다.

### A안: “넓은 쿼리로 재검색”
- service 전용 시드(친절/응대 등)로는 안 걸렸지만,
- 더 포괄적인 쿼리로 다시 검색하면 걸릴 수 있음.

예:
- 1차(서비스 시드): `"친절 응대"` → 0건
- 2차(넓은 쿼리): `"서비스 분위기 음식 가격 만족"` → 몇 건 획득 가능

### B안: “restaurant_id로만 리뷰 N개 가져오기”
- “검색(semantic matching)”이 아니라,
- **그 레스토랑의 리뷰를 그냥 N개 뽑아서**(최근순/임의/전체 중 일부) service 입력으로 넣는 방식입니다.

예:
- service 검색 0건 → `get_restaurant_reviews(restaurant_id)`에서 최근 5개 가져와서 service에 넣기

결과적으로 LLM 입력이 이렇게 바뀝니다:

```json
{
  "service": ["(폴백으로 가져온 리뷰 5개) ..."],
  "price": ["...원래 검색된 리뷰들..."],
  "food": ["...원래 검색된 리뷰들..."]
}
```

## 3) 왜 “0건인 카테고리만” 하냐?

- 0건이 아닌 카테고리는 이미 “그 카테고리와 관련성이 높은 리뷰”를 갖고 있으니 그대로 쓰고,
- **빈 카테고리만** 최소한의 근거라도 넣어줘서 LLM이 “언급이 적다”만 반복하지 않게 하려는 목적입니다.

## 4) 이 방식의 장단점

- **장점**: 빈 카테고리가 사라져서 요약이 좀 더 “말이 되게” 나옴.
- **단점**: 폴백으로 넣은 리뷰가 꼭 service 관련이라는 보장은 약해져서, 프롬프트에 “직접 언급이 없으면 과장하지 말라” 같은 안전장치가 필요함.

원하시는 방향이 A(넓은 쿼리 재검색)인지, B(레스토랑 리뷰 N개를 그냥 채우기)인지에 따라 구현이 달라집니다. 어떤 쪽이 더 맞을까요?

---

아니요. **LLM이 “쿼리를 넓히는” 게 아닙니다.**  
“넓은 쿼리로 재검색”은 **요약 파이프라인(코드)** 에서, 1차 검색이 0건일 때 **우리가 2차로 다른 query_text를 만들어** 벡터 검색을 한 번 더 돌린다는 뜻입니다.

## 누가 넓히나?

- **코드(서버 로직)** 가 넓힙니다. 보통 2가지 방식 중 하나입니다.

### 1) 규칙 기반으로 넓히기 (가장 단순)
예: service가 0건이면, service 시드 대신(또는 추가로)

- `"서비스 친절 응대"` → `"서비스 분위기 음식 가격 만족 리뷰"`  
같이 **미리 정해둔** 더 포괄적인 문자열로 다시 검색합니다.

### 2) LLM을 “쿼리 생성기”로 쓰는 방식 (가능하지만 선택 사항)
LLM에게
- “이 레스토랑의 service 리뷰를 찾기 위한 넓은 검색 쿼리 1~2개 만들어줘”
같이 요청해서 **LLM이 쿼리 문장을 만들어 주고**,  
그 결과를 다시 벡터 검색에 쓰는 방식도 가능합니다.

다만 이건:
- LLM 호출이 한 번 더 들어가서 **비용/지연 증가**
- 쿼리 품질이 들쭉날쭉할 수 있음  
이라서, 보통은 **규칙 기반(1)** 을 먼저 씁니다.

## 정리
- “넓은 쿼리”는 기본적으로 **코드가 정해 만든다**가 정답입니다.
- LLM이 넓히게 할 수도 있지만, 그건 **추가 설계**(선택)입니다.

## 1) 현재 구현된 Dense-only 전략(폴백)

`VectorSearch.query_hybrid_search()` 안에서 아래 상황이면 **Dense-only** (`_query_dense_only`)로 떨어집니다.

- **컬렉션이 단일 벡터 형식**이라 하이브리드가 불가할 때  
- **Sparse 벡터 생성 실패**(sparse 모델 없음/에러)일 때  
- **Qdrant FusionQuery(RRF) 실패**(Sparse 미지원 등)일 때

## 2) Dense-only 결과 품질을 좌우하는 파라미터

Dense-only 폴백에서 중요한 건 두 가지입니다.

- **`limit`**: Dense-only에서 가져오는 최대 hit 수
- **`fallback_min_score`(= Dense-only min_score)**: 이 값보다 점수가 낮으면 버림

지금 구조에서는 하이브리드(RRF)에는 min_score를 적용하지 않고, **Dense-only 폴백에서만** `fallback_min_score`로 컷합니다.

## 3) “Dense를 적극적으로 쓰는” 일반적인 전략(제안)

현재 코드 외에, 보통은 이런 패턴들도 씁니다(아이디어/설계 옵션).

- **Cascading(2단계)**
  - 1차: Dense-only로 넓게(topK 크게, min_score 낮게) 가져와 **recall 확보**
  - 2차: 필요하면 sparse/RRF/재랭크로 **precision 개선**
- **0건 카테고리 구제**
  - 하이브리드가 “성공은 했는데 결과가 0건”인 카테고리에 한해
    - Dense-only를 **min_score=0.0 + limit 크게**로 한 번 더 시도해 “빈 입력”을 줄임
- **동적 튜닝**
  - 쿼리 길이/언어/카테고리별로 `dense_prefetch_limit`, `sparse_prefetch_limit`, `fallback_min_score`를 다르게 두기
  - 예: 짧은 쿼리면 sparse 비중↑ / 긴 문장이면 dense 비중↑

원하시는 “Dense 검색 전략”이 **지금처럼 ‘실패 시 폴백’**인지, 아니면 **‘0건이면 Dense-only 재시도’** 같은 적극적인 정책인지에 따라 접근이 달라집니다.  
원하는 동작을 한 문장으로만 정해주시면(예: “카테고리 0건이면 dense-only로 limit=50 재시도”), 그에 맞춰 적용 위치를 정확히 짚어드릴게요.

“필요하면(2차 rerank/RRF/재랭크)”에서 **필요 여부 판단**은 보통 **1차 검색 결과의 “품질 신호”**로 결정합니다. LLM이 판단하는 게 아니라, **코드가 규칙/지표로 판단**합니다. 실무에서 많이 쓰는 기준들을 정리해 드리면 아래 같습니다.

## 1) 결과 개수(coverage) 기반
- **top-k가 너무 적다**: 예를 들어 `hits < k_min`이면 2차를 켬  
  - 케이스: 카테고리별 검색에서 0~2개만 나오는 상황
- **카테고리별 균형이 깨짐**: service만 0, price/food만 많음 → 그 카테고리만 2차 수행

## 2) 점수 분포/신뢰도 기반 (Dense/Sparse score signals)
(하이브리드는 RRF라 점수가 의미가 약할 수 있지만, Dense-only 단계나 prefetch 단계에서는 유용합니다.)

- **최고 점수가 낮다**: `max_score < threshold` → 결과가 애매하니 2차로 정교화  
- **점수 격차가 작다**: `top1 - topN`이 작으면 랭킹이 불안정 → 2차로 rerank  
- **동점/비슷한 점수 후보가 많다**: tie가 많으면 2차로 더 잘 구분

## 3) 텍스트 기반 간단 필터(cheap heuristic)
후보 리뷰 텍스트를 아주 가볍게 검사해서 “관련성이 약해 보이면” 2차를 켭니다.

- **카테고리 키워드 hit가 거의 없음**  
  - 예: service 후보인데 “친절/응대/직원/서비스” 같은 토큰이 거의 안 나옴  
- **너무 짧은 리뷰만 걸림**  
  - 예: “좋아요”, “굿” 같은 것만 상위에 있으면 2차
- **중복이 심함**  
  - 같은 문장/유사 문장이 상위에 반복 → 2차로 다양성 확보(또는 dedup 후 재검색)

## 4) 쿼리/요청 특성 기반
- **쿼리가 길거나 복합 의도**(예: “데이트 분위기 + 가격 + 주차”)면 2차를 켜서 precision을 올림
- **특정 카테고리(가격 등)** 는 오탐이 잦으면 기본적으로 2차를 켜도록 정책화

## 5) “비용 대비 효과” 기준(운영 관점)
2차는 보통 비용(시간/CPU/추가 호출)이 있으니 아래처럼 정책을 둡니다.

- **특정 조건일 때만** 2차 수행(예: `hits<5` 또는 `max_score<th`)  
- **상위 N개만 rerank** (예: Dense로 50개 가져오고 상위 20개만 rerank)

---

## 지금 코드베이스에 맞춰서 제일 현실적인 판단 기준
현재는 이미 **하이브리드(RRF)** 를 기본으로 쓰고 있고, “2차”라는 단계가 따로 없죠. 이 구조에서 “필요하면”을 만들려면 보통:

- **(A)** 카테고리별 hit가 **0건 또는 너무 적을 때만**  
  → “넓은 쿼리 재검색” 또는 “Dense-only 재시도” 같은 2차를 수행
- **(B)** hit는 있는데 텍스트상 카테고리 키워드가 거의 없을 때만  
  → 2차로 재시도(쿼리 변경 or Dense-only)

이 두 개가 구현도 쉽고, “언급이 적다” 문제(=카테고리 입력이 비는 문제)도 직접적으로 줄입니다.

원하시면, 지금 상황에서 “필요하면” 판단을 **hits 개수 기준으로 할지**, **점수 기준으로 할지**, **키워드 기준으로 할지** 중 어느 쪽이 더 맞는지(성능 vs 정확도 vs 구현 난이도) 기준으로 추천을 좁혀드릴게요.

---

말씀하신 순서도 **논리적으로 맞습니다.** 다만 “항상 Dense→Hybrid를 먼저”가 최선인지는 **목표가 뭐냐(정확도 vs 비용/지연 vs 0건 빈도)**에 따라 달라집니다.

## 왜 “Dense 1차 → Hybrid/RRF 2차 → 그래도 없으면 폴백”이 맞아 보이냐
- **가장 ‘정석’**인 관점에서는,
  1) 1차에서 최대한 후보를 많이 모으고(Recall)  
  2) 2차에서 정교하게 정렬/선별(Precision)  
  3) 그래도 비면(coverage 실패) 마지막에 폴백으로 채움  
  이게 계층적으로 깔끔합니다.

## 그런데 실무에선 “폴백(0건)부터”를 먼저 두기도 함
- 지금 문제가 “0건 카테고리 때문에 LLM이 ‘언급이 적다’로 끝남”이라면,
- **0건을 빠르게 채우는** 게 1차 목표이고,
- 그 목적만 놓고 보면 **0건일 때만 규칙 폴백**이 비용 대비 효과가 커서,
  - Dense→Hybrid 두 번 돌리는 것보다
  - “0건이면 넓은 쿼리 1번 더 / 최근리뷰 N개 채움”이 더 싸고 빠르게 해결될 때가 많습니다.

## 추천은 이렇게 갈라집니다 (한 줄 기준)
- **0건이 자주 난다 / 비용·지연이 민감** → “0건 카테고리만 규칙 폴백”을 **앞에** 두는 게 효율적
- **0건은 가끔 / 품질(정확도)이 KPI** → “Dense 1차 → Hybrid/RRF 2차”를 **먼저** 하고, 그래도 0건이면 폴백

## 둘 다 만족시키는 “현실적인 하이브리드 순서”
카테고리별로 이런 식으로 많이 합니다:

1. **1차 Dense-only** (topK 크게, min_score 낮게/0에 가깝게)로 후보 확보  
2. **2차 Hybrid/RRF** 또는 rerank를 **조건부**로 수행  
   - 조건 예: 후보가 너무 많거나(>X) / 상위 점수 분포가 애매하거나 / 키워드 적합도가 낮거나  
3. **결과가 0건 또는 K_min 미만이면**  
   - 그 카테고리만 **규칙 기반 폴백**(넓은 쿼리 재검색 1회 → 그래도 없으면 최근리뷰 N개)

즉, 질문하신 “그렇게 해야 하는 거 아닌지?”에 답하면:

- **정확도 중심 설계로는 ‘맞다’**(강한 검색/정렬을 먼저 하고, 마지막에 커버리지 폴백).  
- 다만 **현재 문제(0건→언급이 적다)를 빨리 줄이는 목적**만 보면, 폴백을 더 앞에 두는 게 더 실용적일 때도 많습니다.
