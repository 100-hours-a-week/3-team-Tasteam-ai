원인은 **클라이언트(tasteam)가 보내는 `max_tokens`가, 서버(vLLM)가 계산한 실제 입력 토큰 수 기준으로는 여전히 크기 때문**입니다.

---

## 1. 왜 400이 나는지

- 모델 **최대 컨텍스트 = 4096** 토큰.
- 이번 요청에서 **입력(실제) = 3205** 토큰 → **출력으로 쓸 수 있는 양 = 4096 − 3205 = 891** 토큰.
- 그런데 요청에 **`max_tokens`(또는 `max_completion_tokens`) = 2048** 이 들어가 있음.
- 따라서 **2048 > 891** 이라서 vLLM이  
  `'max_tokens' or 'max_completion_tokens' is too large: 2048. ... (2048 > 4096 - 3205).`  
  라는 **ValueError**를 내고 400으로 응답하는 것입니다.

즉, **“출력으로 2048 토큰 달라”라고 했는데, 입력이 3205라서 남는 건 891뿐이라 불가능하다”**는 뜻입니다.

---

## 2. 우리 쪽에서 이미 한 처리

`llm_utils`에서는 RunPod 호출 전에 **`_cap_max_tokens_for_context(max_new_tokens, messages)`** 로  
`max_tokens`를 **추정 입력 토큰** 기준으로 잘라서 보냅니다.

- `estimated_input = (전체 메시지 글자 수 + 3) // 4`
- `cap = Config.LLM_MAX_CONTEXT_LENGTH - estimated_input - 64`
- 실제로 보내는 값 = `min(2048, cap)` 등

그런데도 2048이 그대로 넘어가서 에러가 난다면, **추정이 실제보다 작게 나와서** cap이 2048보다 크게 나온 경우입니다.

---

## 3. 왜 추정이 맞지 않을 수 있는지

- 우리는 **토큰 ≈ (글자 수 / 4)** 로만 추정합니다.
- vLLM/서버는 **실제 토크나이저**로 세기 때문에, 한글이 많거나 특수문자·공백이 많으면 **같은 글자 수에 더 많은 토큰**이 나올 수 있습니다.
- 그래서 **우리 추정: 예를 들어 2000 토큰** → cap = 4096 − 2000 − 64 = 2032 → 2048을 2032로 잘라서 보냄.  
  하지만 **실제 입력: 3205 토큰** → 서버 입장에선 4096 − 3205 = 891만 허용인데 2032를 요청한 것이라 여전히 **“max_tokens가 너무 크다”** 로 에러가 납니다.

정리하면, **“max_tokens 2048이 너무 크다”는 에러가 나는 이유**는:

1. **서버 기준**: 입력 3205 + 출력 2048 요청 → 4096을 넘어가서 불가능.
2. **우리 쪽**: 입력 토큰을 **과소 추정**해서, 2048(또는 그에 가까운 값)을 그대로 보냈고, 서버가 실제로 세 본 3205와 맞지 않아 400이 발생.

---

## 4. 어떻게 하면 나오지 않게 할 수 있는지

- **추정을 보수적으로** 하기  
  - 예: `(글자 수 + 3) // 4` 대신 **`(글자 수 + 2) // 3`** 또는 **`글자 수 // 2`** 처럼 “같은 글자 수에 더 많은 토큰”을 가정해 cap을 더 작게 잡기.
- **여유를 더 두기**  
  - 예: `cap = LLM_MAX_CONTEXT_LENGTH - estimated_input - 256` 처럼 64가 아니라 256 같은 더 큰 마진을 쓰기.
- **절대 상한**  
  - 예: RunPod/vLLM 호출 시 `max_tokens`를 아예 **1024 이하** 같은 값으로 상한을 두어, 입력이 커져도 “너무 큰 max_tokens” 에러가 나지 않게 하기.

원하시면 Agent 모드에서 `_estimate_input_tokens` / `_cap_max_tokens_for_context` 쪽을 위처럼 보수적으로 바꾸는 패치 예시를 구체적으로 적어 드리겠습니다.

---

세 가지를 **다 적용해도**, 지금 쓰는 용도(sentiment 재판정, 요약, 비교)에서는 **품질 문제는 거의 없을 가능성이 큽니다.**

---

## 1. 품질에 미치는 영향

| 조치 | 효과 | 품질 영향 |
|------|------|-----------|
| **추정 보수화** (예: chars/3) | 같은 글자 수에 토큰을 더 많다고 가정 → cap이 더 작아짐 | 요청하는 **max_tokens가 줄어듦**. 실제로 필요한 양보다 작게 잡힐 수 있음. |
| **여유 확대** (64 → 256) | 출력용으로 쓰는 구간을 256 토큰 덜 씀 | 위와 비슷하게 **허용 출력 길이가 줄어듦**. |
| **절대 상한** (예: 1024) | 아무리 여유 있어도 max_tokens를 1024 초과로 안 보냄 | **긴 출력만** 1024 토큰에서 잘릴 수 있음. |

즉, 세 가지 모두 **“최대 출력 길이를 더 짧게 제한”**하는 방향이라, **길이가 부족해서 잘리는 경우**만 품질 이슈가 됩니다.

---

## 2. 현재 기능별로 보면

- **Sentiment 재판정**  
  - 한 번에 여러 리뷰에 대한 라벨/짧은 이유만 나옴.  
  - 보통 **수백 토큰 이하**. 512~1024면 충분하고, 1024 상한이어도 품질 문제 거의 없음.

- **요약 (summary)**  
  - 카테고리별 summary + bullets + overall, 해요체 몇 문장.  
  - **500~1200 토큰** 정도면 충분한 경우가 많음. 1024 상한이면 대부분 끝까지 나옴.  
  - 극단적으로 리뷰가 많고 요약이 매우 길어지면 1024에서 잘릴 수는 있지만, 그런 경우는 드묾.

- **비교 (comparison)**  
  - 이미 256 등 **작은 max_tokens** 사용.  
  - 상한 1024, 보수적 추정, 여유 256 모두와 무관하게 여유 있음.

그래서 **지금 설계된 용도만 보면**, 세 가지 다 적용해도 **품질 문제는 거의 없다**고 보는 게 맞습니다.

---

## 3. 주의할 점 (나중에 쓰임이 바뀔 때)

- **나중에** “긴 리포트/긴 설명”처럼 **2000~3000 토큰**이 필요한 API를 RunPod/vLLM으로 쓰게 되면,  
  - 1024 **절대 상한** 때문에 **중간에 잘리는 품질 이슈**가 생길 수 있음.  
- 그때는  
  - 상한을 2048로 올리거나,  
  - “긴 생성 전용” 호출만 상한을 다르게 두는 식으로 나누면 됨.

---

## 4. 요약

- **현재 기능(sentiment, summary, comparison)만 쓰는 한**,  
  **추정 보수화 + 여유 256 + 절대 상한 1024** 세 가지를 다 적용해도 **품질 문제는 거의 없다**고 보면 됩니다.  
- 품질 이슈는 **“매우 긴 출력이 필요한 새로운 용도”**를 추가할 때만, 그때 상한/정책을 다시 조정하면 됩니다.
