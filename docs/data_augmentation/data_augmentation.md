좋아, 수치가 나왔으니 **“몇 개 샘플(= 음식점 단위 요약 학습 샘플)”을 만들 수 있나**를 현실적으로 계산해볼게.

* 음식점: **47개**
* 총 리뷰: **20,480개**
* 음식점당 평균 리뷰 수:
  ( 20,480 ÷ 47 \approx 435.7 ) → **약 436개/식당**

즉, “식당 수는 적지만, 각 식당에 리뷰가 아주 많다” 케이스야.
이 경우는 **식당을 늘리지 않고도 ‘식당 내부 분할’로 샘플을 꽤 많이 뽑을 수 있어.**

---

## 1) 가장 깔끔한 샘플 생성 방식: 슬라이딩 윈도우

한 식당의 리뷰를 시간순으로 정렬하고, “리뷰 N개 묶음”을 여러 개 만든 다음 각 묶음마다 요약 1개를 만드는 방식.

### 계산 공식

* 윈도우 크기 = `W` (예: 30개 리뷰)
* 스트라이드 = `S` (예: 15개씩 밀기)
* 한 식당에서 나오는 샘플 수 ≈ ( \lfloor (R - W)/S \rfloor + 1 )

여기서 평균 R=436을 넣어보면:

### 예시 A: W=30, S=15

* ( (436-30)/15 = 406/15 \approx 27.06 )
* +1 해서 **약 28개/식당**
* 47개 식당이면 **28×47 = 1,316 샘플**

### 예시 B: W=50, S=25

* ( (436-50)/25 = 386/25 \approx 15.44 )
* +1 → **약 16개/식당**
* 전체 **16×47 = 752 샘플**

### 예시 C: W=20, S=10

* ( (436-20)/10 = 416/10 = 41.6 )
* +1 → **약 42개/식당**
* 전체 **42×47 = 1,974 샘플**

➡️ **한 가지 윈도우 세팅만 써도 700~2,000 샘플은 현실적으로 확보 가능**해.

---

## 2) “5,000 샘플”도 가능하냐?

가능한데, 조건은 이거야:

### 방법: 여러 윈도우 크기를 섞는다

예를 들어 **3가지**만 섞어도:

* W=20,S=10 → 1,974
* W=30,S=15 → 1,316
* W=50,S=25 → 752

합계 = **4,042 샘플**

여기서 “전체 리뷰(436개 전부)” 요약 1개씩을 식당당 추가하면:

* +47 → **4,089**

그리고 여기에 **감성 분할**(긍정/부정만 따로 묶기)이나 **최근 N개 요약** 같은 변형을 조금만 추가하면 **5,000 근처**까지는 충분히 도달해.

---

## 3) 단, “샘플 수”보다 더 중요한 리스크

식당 수가 47개로 적으니, 윈도우로 샘플을 많이 만들수록 이런 문제가 생겨:

* **입력이 서로 너무 비슷해짐**(중복/누수)
* “실제 일반화”가 아니라 “그 47개 식당에 과적응”될 수 있음

그래서 5,000을 만들더라도 “유효한 5,000”으로 만들려면:

* **스트라이드 너무 촘촘하게 하지 않기** (S를 W의 1/2 정도로)
* **중복 제거**(입력 유사도/키워드 겹침 기준)
* **평가셋은 식당 단위로 홀드아웃**(중요!)
  → 같은 식당에서 train/test가 섞이면 평가가 망가져

---

## 결론

* 평균 436리뷰/식당이면
  **슬라이딩 윈도우만으로도 1,000~4,000 샘플은 무난**
* **윈도우 크기 2~3개 + 감성 분할/기간 분할**을 더하면
  **5,000 샘플도 가능**
* 다만 **평가는 반드시 “식당 단위”로 분리**해야 신뢰할 수 있음

원하면 내가 너한테 “딱 5,000개를 목표로”

* 윈도우 크기/스트라이드 조합
* 생성 샘플 수 산출
* 중복 제거 규칙
* 식당 단위 train/val/test 분할 방식
  까지 한 번에 설계안으로 만들어줄게.

---

어떻게 나누면 되나?
1) 제일 권장: 식당 단위 Split

Train: 37개 식당

Val: 5개 식당

Test: 5개 식당

47개면 이 정도가 가장 무난해.
(대략 80/10/10 느낌)

이렇게 하면 test는 “완전히 처음 보는 식당”이라서 평가가 의미가 생겨.

그럼 test 샘플 수는 얼마나 되나?

식당당 샘플 수가 윈도우 설정에 따라 달라지지만, 예를 들어:

W=30, S=15로 하면 식당당 약 28 샘플

Test 5개 식당이면 대략 5×28 = 140 샘플 정도

이 정도면 테스트로 충분히 쓸 수 있어.

2) 만약 식당마다 리뷰 수 편차가 크면?

리뷰 많은 식당이 test로 몰리면 test 샘플이 과하게 많아지고,
적은 식당이 몰리면 너무 적어질 수 있어.

그래서 분할할 때는:

식당별 리뷰 수(또는 샘플 수)를 보고

비슷하게 섞이도록 나누는 게 좋다. (stratified split 느낌)

## 실행 스크립트

위 설계를 구현한 스크립트: **`scripts/data_augmentation.py`**

- **입력**: `tasteam_app_all_review_data.json` (또는 `--input`으로 지정). `reviews` 배열에 `restaurant_id`, `content`, `created_at` 필요.
- **동작**: 식당별 리뷰를 시간순 정렬 → 슬라이딩 윈도우(W,S) 여러 조합 적용 → 식당 단위 stratified train/val/test 분할 → 샘플별로 `split` 부여.
- **출력**: `--out-dir`(기본 `data_augmentation_output/`) 아래 `train.json`, `val.json`, `test.json`, `stats.json`.

```bash
# 기본 실행 (윈도우 30/15, 50/25, 20/10 + 식당 비율 80/10/10)
python scripts/data_augmentation.py

# 식당 전체 리뷰 1개 샘플씩 추가
python scripts/data_augmentation.py --add-full

# 입출력·윈도우·비율 지정
python scripts/data_augmentation.py --input tasteam_app_all_review_data.json --out-dir out --window 30 15 50 25 --add-full --train-ratio 0.8 --val-ratio 0.1 --test-ratio 0.1 --seed 42
```
