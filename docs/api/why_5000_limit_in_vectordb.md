
네, 맞습니다. **리소스(메모리·연산·시간)를 줄이려고** `limit`으로 적당한 수만 쓰는 설계로 보는 게 맞습니다.

- **메모리**: 리뷰 텍스트를 전부 올리면 수만~수십만 건일 때 메모리 사용이 커짐.
- **연산/시간**: Kiwi(또는 Spark)로 비율 계산하는 양이 늘어나면 요청당 지연이 길어짐.
- **통계적 충분성**: “전체 평균” 추정에는 수천 건만 있어도 대개 충분하고, 5만 건으로 올려도 결과 차이가 크지 않은 경우가 많음.

그래서 **5000 같은 limit**은 “전체 평균 품질은 유지하면서 리소스는 적당히” 쓰자는 타협점으로 이해하면 됩니다.  
리소스 여유가 있고 정말 전부 써야 하는 요구가 있으면 limit을 키우거나 전체 스크롤을 쓰면 되고, 보통은 지금처럼 limit 두는 쪽이 맞습니다.
