그 loss 값( **train 1.7759 / eval 1.969** )만 보면 “학습이 망했다” 느낌은 아니야. 오히려 **겉보기엔 정상 범위**일 수 있어.
근데 **지금 너가 겪는 문제(포맷 흔들림/스타일 불안정)** 는 **loss로 잘 안 잡히는 유형**이라서, 저 숫자가 “괜찮다”를 보장하진 않아.

핵심만 정리하면:

## 1) 지금 loss 갭은 크지 않다 → 과적합 신호는 약함

차이 = 1.969 − 1.7759 = **0.1931**.
이 정도면 “train만 잘 맞추고 eval 망함” 같은 전형적 과적합 패턴은 아님.

## 2) 그런데도 문제가 생길 수 있는 이유: **loss는 ‘정답 토큰’만 본다**

너는 사실상 **KD(SFT)로 teacher 출력 텍스트를 따라 쓰게** 만들고 있지?

이때 teacher가 섞여 있으면,

* 어떤 샘플은 4o mini 스타일
* 어떤 샘플은 7B 스타일

이런 **분포 혼합**이 생기는데, 모델은 그냥 “각 샘플의 다음 토큰 예측”만 잘하면 loss가 내려가.
즉, **스타일이 들쭉날쭉해도 loss는 낮게 나올 수 있어.**

특히 JSON 태스크는 더 심해:

* 괄호/따옴표/필드명 1~2개 틀려도 loss 변화는 작을 수 있는데
* 결과물은 **파싱 불가**로 바로 0점이 되잖아.

## 3) 그래서 “loss 정상”인데 “출력은 불안정”이 흔함

정리하면 지금 상태는 딱 이 케이스 가능성이 큼:

* loss: 무난 (teacher 텍스트를 평균적으로 잘 모방)
* output: 불안정 (teacher 스타일/스키마가 섞여서 일관성이 깨짐)

## 4) 이럴 때 확인해야 하는 3가지(바로 효과 큼)

1. **JSON 파싱 성공률** (eval 샘플에서 %로)
2. **스키마 정확도** (필수 키 존재/타입 맞는지)
3. **길이/포맷 drift** (teacher별로 평균 길이, 키 분포가 다른지)

loss보다 이게 훨씬 “진짜 성능”을 말해줘.

## 5) 결론: loss가 저 정도면 “모델 크기(0.5B)가 너무 작아서”라고 단정하긴 어려움

오히려 지금은 우선순위가:

* **teacher 분포를 4o mini로 단일화**
* **JSON 스키마 강제(후처리/constraint/검증 기반 리라이트)**

이 두 개를 먼저 하면, 같은 loss여도 **체감 품질이 확 올라가는** 경우가 많아.

원하면, 네 현재 `val_predictions.json` 같은 샘플 20개만( pred/ref ) 붙여줘.
내가 **“loss는 괜찮은데 왜 output이 망가지는지”**를 케이스별로 딱 갈라서 원인(혼합 teacher vs decoding vs 데이터 포맷 vs tokenizer/템플릿)을 바로 짚어줄게.

# 1️⃣ 왜 이런 일이 생기냐 (LLM 학습 구조 때문)

LLM은 사실 이렇게 학습한다.

```
P(token_t | token_1 ... token_{t-1})
```

즉 **한 토큰씩 맞추는 문제**다.

하지만 우리가 원하는 것은

```
correct structured output
```

이다.

예:

```
{
  "service": {...},
  "price": {...},
  "food": {...}
}
```

이건 사실

* 구조
* 필드 순서
* 문체
* 길이
* 내용

을 **전체적으로 맞춰야 하는 task**다.

근데 학습은 **토큰 단위**다.

그래서 이런 일이 생긴다.

---

# 2️⃣ teacher가 섞이면 더 심해진다

지금 데이터:

```
4o mini
+
7B
```

이러면 모델이 배우는 것은

```
JSON style A
JSON style B
bullet style C
language mix
```

이다.

그래서 학습은 이렇게 된다.

```
다음 토큰 예측 평균 최적화
```

예:

```
"service":
다음 토큰은

{
  "summary":
또는
{
  "description":
또는
Overall rating
```

이런 식으로 **여러 가능성이 생긴다.**

loss 입장에서는:

```
각 샘플에서는 맞음
```

그래서 loss는 낮다.

하지만 inference에서는

```
랜덤하게 스타일이 섞임
```

---

# 3️⃣ 특히 작은 모델에서 더 심하다

0.5B는 capacity가 작아서

```
teacher distribution을 정확히 분리해서 기억
```

하지 못한다.

그래서 내부적으로

```
평균 스타일
```

을 배운다.

이걸 **mode averaging**이라고 한다.

그래서 결과는

```
이상한 hybrid output
```

이 된다.

예:

```
{
  "service": ...
}

Overall rating: ...
```

같은 것.

---

# 4️⃣ 그래서 loss와 실제 성능이 다를 수 있다

예를 들어:

teacher 1

```
{
  "service": ...
}
```

teacher 2

```
Service Summary:
...
```

모델이 이렇게 출력하면

```
{
  "service":
Service Summary:
```

loss는 크게 증가하지 않는다.

하지만 output은 **완전히 망가진다.**
